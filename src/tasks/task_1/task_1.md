# Задача для 1 семестра

#### 1. Суть кейса (что строим)

1. **Цель системы (миссия)**
2. **Контекст (среда и внешние системы)**  
   2.1. Пользователи - описать поведение
   2.2. Исполнители - описать поведение
   2.3. Интеграции (на выбор команды) - разработать
3. **Границы SoI**  
   3.1. Внутри: сервисы, база, модули, уведомления, отчётность, админка.  
   3.2. Снаружи: IdP/SSO, почтовый сервер, внешние мониторинги, BI/аналитика (опционально).

#### 2. Роли и сценарии (CONOPS)

1. Роли  - описать роли по модели CONOPS
2. Минимум 5 сценариев (команда обязана описать в CONOPS)

#### 3. Что нужно сдать - Пакет артефактов (обязательный минимум)

1. **Контекст и границы**  
   1.1. Список внешних сущностей + потоки (текстовая “контекстная диаграмма”: кто/что ↔ какие данные/события).
2. **CONOPS/OPSCON**  
   2.1. 5 сценариев + исключения + режимы деградации (например, недоступен SSO/почта).
3. **Требования**  
   3.1. Stakeholder requirements: 10–15 шт.  
   3.2. System requirements: 25–40 шт. (**атомарные, проверяемые**)  
   3.3. NFR/качество: минимум 10 требований, привязанных к метрикам.
4. **Архитектура**  
   4.1. Декомпозиция на компоненты/сервисы + обязанности.  
   4.2. 3–5 ключевых архитектурных решений с trade-off обоснованием.
5. **Интерфейсы (ICD-lite)**  
   5.1. 3–6 контрактов: API или события (в виде таблиц: endpoint/event, поля, ошибки, версии).
6. **Риски**  
   6.1. Реестр рисков: минимум 8 (причина→событие→последствие→вероятность/ущерб→меры→остаточный риск).
7. **V&V**  
   7.1. Матрица трассируемости: requirement → метод проверки (тест/анализ/инспекция/демо) → артефакт/кейс.  
   7.2. Минимум 12 тест-кейсов (описанием, не кодом).
